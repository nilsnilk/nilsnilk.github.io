AR ENVIRONMENT FOR DEMENTIA PATIENTS

PROJECT SUMMARY

This project was part of a course at Umeå University which mainly regarded ubiqutous computing and smart environments. Our group, which consisted of students that studied cognitive science, computer science, interaction design and occupational therapy, were given the task to develop a smart environment for dementia patients using augumented reality.
We all agreed that augumented reality could potentially cause more confusion for a user suffering from dementia, so we decided that something more passive and non-invasive was the way to go. We also agreed that it would be best if the system would be a part of the user's environment, since a wearable easily could be forgotten by someone with dementia.
We learned that one of the biggest problems for dementia patients is structure; when to eat, sleep, when the home care personell will show up etcetera. Dementia patients often feel lost, which makes them worried a lot. This can often result in false alarms which the home care staff must respond to. Our solution was a interactive calendar which could be accessed remotely by the home care staff. This calendar would contain a schedule for each day, with the purpose to introduce structure, and thus calm the user.
The AR-part of this system consisted of a kinect, which would keep track of the patients position and orientation in the environment. This information would then be used to direct a projector mounted on a robotic arm in the ceiling, which if required would project information about upcoming scheduled events in the calendar. The system would also have a neural network which would be trained to detect different types of worry behaviours using the kinect. If the system determine that the user is worried, it would introduce some kind of element such that would calm the user. This element could be a favourite song, or a message that is projected on a surface in the field of view of the user.

SCHNELLE BRILLEN APP

PROJECT SUMMARY

This project was part of a course called "Produktutveckling i medieteknik med metoden Design-Build-Test" at Umeå University, where the course participants were divided into teams. Each team were then given the task to design, build and test a concept for a company.
Our group was given the task to develop an app for a web-based sunglasses store in the netherlands called "schnellebrillen.nl". This company has a product range that consists of what they call "fast glasses", which is a slang expression that roughly refers to sunglasses that looks fast, i.e is aerodynamic.
We decided to create a IOS and a android app which could determine wether a pair of sunglasses is "fast" or not, this would be determined with the help of a neural network. In addition, the app were to have a store where the user can browse the store's products.
We ended up with a working prototype with three views; a home view (visible in the image to the right), a camera view and a store. the pictures taken by the camera is sent to two different tensorflow-models, where the first one finds the sunglasses, and then sends a cropped version of the picture to the other tensorflow-model, which determines wether the sunglasses are "fast" or not. In order to train these models, we had to create two large datasets, which consisted of pictures depicting people with "fast" and "slow" glasses respectively. these data sets consisted of around 2000 pictures each. The app will create visual indicators in the picture if and where sunglasses are found, and wether they are "fast" or not.

COMMUNITY FOR WEATHER TALK

PROJECT SUMMARY

This project was part of a course at Umeå University. The objective was to create a web-based application based on the model-view-controller structure. In addition to this, the application had to use a database.
We decided to create a community which were similar in style to reddit, where the users only were allowed to talk about the weather. We came up with a few different sub-forums which were based on the different components of the weather; temperature, pressure, rain/snowfall etcetera. We also wanted to show the weather in real time on the page.
We created the page in ASP.NET (Visual studio), and used Umeå University's own weather station API to show the weather. We made the application database-first with entity framework.

MAPDRAW

PROJECT SUMMARY

This project was part of a course at Umeå University, where the objective was to create a prototype for a mobile application.
Me and the rest of my group decided to make a GPS-position based game where the users could "draw" on a map by walking around. The user would push a button which left a trail behind them on the map when they moved. We wanted the user to be able to change color and thickness of the trail. Lastly, we wanted that other users could see the drawings on their devices as they were made in real-time.
In order to do this we first decided to create a client for IOS with Swift and a server built with Node.js, MongoDB, Mongoose and socket.io. Even though the server was ready to go we decided to scrap the real-time solution; we determined that we didn't have the time needed in order to implement the sockets on the client-side. In the end we ended up with a solution where the user can choose whether they want to upload the drawing or not. In the socket.io-based solution required the client to continually send its position to the server. The solution we settled with only needed to post the drawing once to the database, which was fairly easy to implement with MongoDB and Mongoose.

CLEANING APP USING LORA

PROJECT SUMMARY

This project was part of a course at Umeå University where you were given access to a network of LoRa-sensors that were spread across campus. The goal of the project was to use create an application that utilized these sensors.
The sensors were able to register if people were in its proximity using a infra-red sensor. Me and the rest of my group came up with an app which used this together with the current weather (provided by the Umeå University's weather station) determined if a space with a sensor needed cleaning. The idea was that the spaces would become dirty faster under wet weather conditions.
Our solution consisted of an working prototype with a client build with React Native and a server built with Node.js, MongoDB and Mongoose. The server retrieved data from the sensors every 30 minutes. Then it calculated a "dirtyness-index" for each space based on the amount of detected people and the weather, which it then stored in a database. The client could then retrieve the "dirtyness-index" from the database. In addition to this, the client have the capability to update the database in order to mark spaces as clean. This allows the user to reset the "dirtyness-index" of a space if they have cleaned it.
